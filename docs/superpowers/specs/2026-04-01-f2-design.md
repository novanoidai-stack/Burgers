# F2 Design Spec — Cerebro IA
**Fecha**: 2026-04-01
**Estado**: Aprobado por usuario
**Autor**: Brainstorming session

---

## Visión

F2 convierte la infraestructura base de F1 en un asistente comercial funcional. Un cliente llama al +1(938)465-3399 o escribe por WhatsApp, la IA toma su pedido en lenguaje natural, cobra con Stripe, y guarda la orden en Supabase como `paid`.

---

## Arquitectura General

```
ENTRADA
────────────────────────────────────────────
[Twilio SIP]  →  audio μ-law 8kHz
                      ↓
              [Deepgram STT]  →  transcript
                      ↓
[WhatsApp]    →  texto                ↓
                      ↓        [Input Normalizer]
                      └──────────────┘
                                ↓
                    {channel, sessionId, tenantId, text}

CEREBRO
────────────────────────────────────────────
                    [Session Manager]  ←→  Supabase
                    (estado conversación: items, fase)
                                ↓
                    [LLM Orchestrator] (OpenRouter)
                    - System prompt dinámico con menú de DB
                    - Function calling para estructurar pedido
                                ↓
              ┌─────────────────────────────┐
              │  add_item / remove_item     │
              │  confirm_order              │
              │  ask_clarification          │
              └─────────────────────────────┘

SALIDA
────────────────────────────────────────────
                    [Output Router]
                    ↙              ↘
          voz: ElevenLabs TTS    texto: WhatsApp reply
          → Twilio audio stream

PAGO
────────────────────────────────────────────
                    [Stripe: crea Payment Link]
                    → se envía al cliente (audio o texto)
                    ↓
                    [Stripe Webhook: payment.succeeded]
                    ↓
                    Order en DB → status: PAID
```

**Principio clave**: Voz y WhatsApp convergen en el mismo `Input Normalizer`. A partir de ahí, el LLM no sabe ni le importa de qué canal vino el mensaje. La respuesta sale por el canal correcto al final.

---

## State Machine

```
GREETING
  ↓ "Hola, soy la IA de El Mesón, ¿qué te pongo?"
TAKING_ORDER
  ↓ acumula items con function calling
  ↓ ←─────────────────────────────┐
CONFIRMING                        │ "quiero cambiar algo"
  ↓ lee resumen + total           │
  ↓ cliente confirma  ────────────┘
AWAITING_PAYMENT
  ↓ [voz] IA pide número → envía link por WhatsApp/SMS
  ↓ [WhatsApp] IA envía link directo en el chat
  ↓ cliente paga
PAID ✅ → orden en DB, IA confirma tiempo estimado
DONE
```

**Casos especiales**:
- Cliente cuelga antes de pagar → orden queda `abandoned` tras 10 min
- Stripe webhook nunca llega → reintento automático x3, luego `payment_failed`
- Cliente pide producto que no existe → LLM responde con alternativa del menú
- CONFIRMING → TAKING_ORDER: cliente quiere cambiar algo antes de pagar

## Componentes

| Módulo | Archivo | Responsabilidad |
|--------|---------|----------------|
| `InputNormalizer` | `src/backend/services/inputNormalizer.ts` | Convierte transcript/texto → mensaje normalizado `{channel, sessionId, tenantId, text}` |
| `SessionManager` | `src/backend/services/sessionManager.ts` | Guarda/recupera estado de conversación en Supabase |
| `LLMOrchestrator` | `src/backend/services/llmOrchestrator.ts` | OpenRouter + system prompt dinámico + function calling |
| `OutputRouter` | `src/backend/services/outputRouter.ts` | Decide si responde por TTS (voz) o texto (WhatsApp) |
| `PaymentService` | `src/backend/services/paymentService.ts` | Crea Stripe Payment Links, valida webhooks |

## LLM + Function Calling

### System Prompt (estructura)

```
Eres el asistente de voz de El Mesón, una smashburger de Pamplona.
Tu único trabajo: tomar pedidos de forma natural y amigable.

REGLAS:
- Solo puedes ofrecer productos del menú
- No inventas precios ni descuentos
- Si el cliente pregunta algo fuera del menú, redirige educadamente
- Hablas en español, tono cercano pero profesional

MENÚ ACTUAL: [inyectado dinámicamente desde DB en cada llamada]
```

### Function Calling Tools

```typescript
add_item(product_id: string, quantity: number, modifiers?: string[])
// "ponme una Super Smash Bros sin pepinillos"

remove_item(product_id: string)
// "quita la Chupacabras"

get_order_summary()
// IA lee el pedido completo antes de confirmar

confirm_order()
// cliente dice "sí, eso es todo" → dispara flujo de pago

request_phone_number()
// solo en canal voz, para enviar el link de Stripe
```

## Flujo de Pago (Stripe)

```
1. confirm_order() ejecutado por LLM
2. PaymentService crea Stripe Payment Link
   - amount: total del pedido en céntimos
   - metadata: { order_id, tenant_id, session_id }
   - expires_in: 15 minutos
3. Link enviado al cliente
   - Voz → WhatsApp/SMS al número recogido en request_phone_number()
   - WhatsApp → mensaje directo en el chat
4. Cliente paga en Stripe
5. Stripe → POST /webhooks/stripe
   - Valida firma con STRIPE_WEBHOOK_SECRET
   - Extrae order_id del metadata
   - Actualiza orden: status = 'paid'
   - Idempotencia: mismo payment_intent_id no duplica orden
6. IA confirma: "¡Listo! Tu pedido está en camino 🍔"
```

**Timeout**: orden sin pago en 15 min → status `abandoned`

## Base de Datos (nuevas tablas en `restaurant_001`)

```sql
products
  id UUID PK
  name TEXT NOT NULL
  price INTEGER NOT NULL  -- en céntimos
  category TEXT           -- 'burger' | 'racion' | 'postre' | 'bebida'
  description TEXT
  available BOOLEAN DEFAULT true
  created_at TIMESTAMPTZ

order_items
  id UUID PK
  order_id UUID FK → orders.id
  product_id UUID FK → products.id
  quantity INTEGER NOT NULL
  modifiers TEXT[]         -- ["sin pepinillos", "extra queso"]
  unit_price INTEGER NOT NULL
  created_at TIMESTAMPTZ

conversations
  id UUID PK
  session_id TEXT UNIQUE NOT NULL
  channel TEXT NOT NULL    -- 'voice' | 'whatsapp'
  phone_number TEXT
  status TEXT NOT NULL     -- 'greeting' | 'taking_order' | 'confirming' |
                           --  'awaiting_payment' | 'paid' | 'done' | 'abandoned'
  order_id UUID FK → orders.id
  stripe_payment_link TEXT
  stripe_payment_intent_id TEXT
  created_at TIMESTAMPTZ
  updated_at TIMESTAMPTZ
```

## Extensibilidad (diseño intencionado)

El `Input Normalizer` actúa como puente universal. Añadir un canal nuevo = nuevo adaptador que convierte esa entrada al formato interno `{channel, sessionId, tenantId, text}`. El LLM, Stripe y la DB no cambian.

```
FUTURO (sin tocar el núcleo):
Web widget       ──┐
App móvil        ──┤
Reservas         ──┤──→ Input Normalizer ──→ LLM ──→ DB ──→ Stripe
Comanda en mesa  ──┤
Telegram/Signal  ──┘
```

Extensiones futuras requerirán:
- Nuevo adaptador de entrada (ej. `TableOrderNormalizer`)
- Nuevas tools de function calling (`make_reservation`, `order_to_table`)
- Nuevos estados en state machine
- Nuevos campos en DB (no breaking changes)
